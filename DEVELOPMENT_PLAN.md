# TGO 微信私域 AI 客服开发计划

## 1. 文档信息

- 项目名称：TGO 微信私域 AI 客服增强版
- 基础项目：[tgoai/tgo](https://github.com/tgoai/tgo)
- 建设方式：Fork TGO 后，在其单仓库架构内进行二次开发
- 默认渠道：企业微信官方“微信客服” `wecom_kf`
- 默认部署形态：单企业、单租户、私有化部署
- 计划周期：3 人核心团队 10～12 周；单人全栈开发 14～18 周
- 当前状态：第一阶段安全核心已合并到个人 Fork 的 `main`；真实渠道、第三方 Provider 和业务系统联调待完成

### 1.1 当前实施进度（2026-07-17）

- Git 验证基线固定在上游 commit `995da4577f6f91edb87d0f56fc9ea4c129f1a4eb`；个人 Fork 为 `https://github.com/jliu95746-dotcom/tgo`，本轮安全核心开发提交为 `49626edbbd6a47a5d970200f92966d91a5f06562`。

## 2. 项目结论

本项目不重新建设一套独立客服平台，而是直接以 TGO 为产品主体，复用其渠道接入、访客与会话、AI Agent、Workflow、RAG、插件运行时、人工客服、管理后台、PostgreSQL、Redis、Celery 和 WuKongIM。

第一阶段只补齐以下能力：

1. 微信客服文字链路的稳定性验证和必要修复。
2. 微信语音和图片消息的完整接收与异步处理。
3. ASR 语音转写、OCR 文字提取和 VLM 图片理解。
4. 意图识别、实体提取、风险分级和置信度路由。
5. 产品知识、售后政策和客服 SOP 的 RAG 问答。
6. 订单、物流、会员和 CRM 的只读工具接入。
7. 低置信度、高风险场景和处理失败时的人工转接。
8. 客服工作台中的多模态结果、意图和处理状态展示。
9. 测试、审计、监控、备份、恢复和数据安全措施。

## 5. 总体架构

```text
微信客服 wecom_kf
→ Nginx / HTTPS
→ tgo-platform：验签、解密、同步、去重、快速落库
→ Redis / Celery：异步任务
→ 多模态处理模块：ASR、OCR、VLM
→ tgo-ai：意图识别、实体提取、安全策略
→ tgo-workflow：业务路由
   ├── tgo-rag：产品、售后、FAQ、SOP
   ├── tgo-plugin-runtime：订单、物流、会员、CRM
   └── tgo-api / tgo-web：人工客服
→ tgo-platform：回复微信
```

架构原则：

- TGO 是系统主体，不建立平行客服系统。
- 优先扩展现有模块，第一版不拆分不必要的微服务。
- 微信回调只负责验签、解密、落库和任务投递，不等待大模型处理。
- 所有业务工具遵循白名单、最小权限、参数校验和完整审计。
- 多模态或模型处理失败时必须提示重发或转人工，禁止静默丢失。
- 上游代码采用固定 commit 和固定镜像 digest，不直接追随浮动 `main` 或 `latest`。

## 7. 微信消息处理流程

### 7.1 通用处理步骤

1. `tgo-platform` 接收微信回调。
2. 校验签名并解密消息。
3. 提取消息 ID、客户 ID、客服 ID、消息类型和 `media_id`。
4. 使用微信消息 ID 和平台 ID 建立幂等约束。
5. 原始消息落库，立即向微信返回成功响应。
6. 将消息提交到 Celery 异步队列。
7. 文字消息直接标准化；语音和图片进入多模态处理。
8. 将标准化文本、多模态结果和最近会话发送给 `tgo-ai`。
9. `tgo-ai` 返回意图、实体、置信度和风险等级。
10. `tgo-workflow` 决定 RAG、业务工具、追问或转人工。
11. 回复前执行安全检查。
12. `tgo-platform` 将结果发送到微信。
13. 全流程记录 `request_id`、`message_id`、处理状态和审计信息。

### 7.2 统一消息结构

```json
{
  "message_id": "wechat-message-id",
  "channel": "wecom_kf",
  "message_type": "voice",
  "customer_id": "external-user-id",
  "conversation_id": "conversation-id",
  "text": "我想查一下订单什么时候到",
  "derived": {
    "transcript": "我想查一下订单什么时候到",
    "ocr_text": "",
    "vision_summary": ""
  },
  "intent": {
    "name": "logistics_query",
    "confidence": 0.93,
    "taxonomy_version": "v1"
  },
  "entities": {
    "order_no": null,
    "product_name": null
  },
  "route": "business_tool",
  "need_human": false
}
```

## 8. 多模态处理

第一版不新增独立 `tgo-media` 微服务，而是在 TGO 现有 Python 和 Celery 体系中增加媒体处理模块及 Worker。

### 8.1 语音处理

1. 收到 `media_id` 后立即创建异步任务。
2. 下载原始语音并计算 SHA-256。
3. 校验格式、文件大小和时长。
4. 必要时使用 FFmpeg 转为 ASR 支持的 WAV/PCM。
5. 调用可替换的 ASR Provider。
6. 保存转写文本、时间戳、置信度和模型版本。
7. 将转写文本与最近会话共同用于意图识别。
8. 失败时按策略重试；最终失败则提示重发或转人工。

### 8.2 图片处理

1. 下载图片并校验 MIME、大小和完整性。
2. 加密存入对象存储，外部访问使用短期签名 URL。
3. OCR 提取订单号、商品名、错误信息等文字。
4. VLM 识别商品、物流截图、破损、支付失败等场景。
5. 对手机号、地址、身份证和支付信息进行脱敏。
6. 合并用户文字、OCR、图片摘要和最近对话。
7. 执行意图识别、实体提取和风险判断。
8. 无法理解或涉及敏感凭证时转人工。

OCR 和图片中的所有文字均视为不可信输入，不得直接成为系统提示词或直接触发业务工具。

## 9. 意图识别和路由

第一版使用“规则优先 + 大模型结构化 JSON + 置信度路由”，暂不训练专用分类模型。

初始意图：

- 产品咨询
- 价格和优惠
- 下单帮助
- 订单查询
- 物流查询
- 支付问题
- 售后故障
- 退款退货咨询
- 投诉
- 线索客户
- 转人工
- 无法识别

分类结果必须包含：

- `intent`
- `confidence`
- `entities`
- `risk_level`
- `recommended_route`
- `need_human`
- `taxonomy_version`

初始路由规则：

| 条件 | 处理方式 |
|---|---|
| `confidence >= 0.85` 且为 FAQ/只读查询 | 自动回复或调用只读工具 |
| `0.60 <= confidence < 0.85` | 向客户确认一次 |
| `confidence < 0.60` | 转人工 |
| 投诉、退款执行、改价、账户变更 | 强制人工审核 |
| 媒体识别失败 | 提示重发或转人工 |
| 连续两次无法识别 | 自动转人工 |

正式阈值必须通过脱敏历史会话评测确定，不能只依赖模型自报置信度。

## 10. Workflow 设计

```text
消息进入
→ 媒体是否处理成功
→ 敏感信息检测
→ 意图识别
→ 置信度判断
   ├── 产品/售后咨询 → tgo-rag
   ├── 订单/物流查询 → Plugin Tool
   ├── 信息不完整 → 追问确认
   ├── 低置信度 → 人工客服
   └── 投诉/高风险 → 人工客服
→ 回复安全检查
→ 发送微信
```

## 11. RAG 知识库

直接使用 `tgo-rag`，导入：

- 产品说明
- 价格和活动规则
- 售后政策
- 退款退货规则
- 物流时效
- 常见问题
- 客服话术
- 人工转接 SOP

每份知识资料应包含：

- 产品线
- 适用渠道
- 生效时间
- 失效时间
- 负责人
- 文档版本
- 是否允许自动回复

知识更新必须经过审核，不允许直接使用未确认的临时文档覆盖生产知识库。

## 12. 业务系统工具

第一版通过 `tgo-plugin-runtime`、MCP 或 OpenAPI 工具接入，不单独建设业务网关服务。

建议工具：

- `OrderQueryTool`
- `LogisticsQueryTool`
- `CustomerProfileTool`
- `MembershipQueryTool`
- `ProductInventoryTool`

第一阶段仅允许只读操作。退款、取消订单、改地址、改价等写操作暂不开放给 AI。

所有工具必须：

- 使用最小权限服务账号。
- 校验客户身份与订单归属。
- 校验参数类型和允许范围。
- 设置超时、重试和熔断。
- 记录调用人、会话、参数摘要、结果和耗时。
- 对日志中的个人信息进行脱敏。

## 13. 数据扩展

复用 TGO 现有用户、访客、会话、消息和客服表，通过对应服务的 Alembic 迁移新增：

| 表 | 用途 |
|---|---|
| `message_media` | 媒体类型、存储位置、校验值和删除时间 |
| `media_processing_jobs` | 任务状态、重试次数和错误信息 |
| `media_analysis_results` | ASR、OCR、图片摘要和模型版本 |
| `message_intent_results` | 意图、实体、置信度、风险和路由 |
| `business_tool_call_logs` | 订单、物流等工具调用审计 |
| `conversation_handoff_events` | 转人工原因、客服和处理时间 |
| `prompt_versions` | 分类、路由和回复提示词版本 |
| `evaluation_results` | 离线评测和回归结果 |

原始媒体保留期必须可配置。删除流程应覆盖数据库、对象存储和备份生命周期。

## 18. 部署方案

### 18.1 环境建议

- 开发：WSL2/Ubuntu，4 核、16 GB。
- 测试：Linux，8 核、16 GB。
- 生产起步：Linux，8 核、32 GB。
- MVP 使用 TGO 现有 Docker Compose。
- 数据库和对象存储必须有独立备份。
- 只有在需要多实例、跨机器高可用时再迁移 Kubernetes。

### 18.2 发布流程

```text
Feature Branch
→ 单元和契约测试
→ Lint、类型检查、密钥扫描
→ 构建固定版本镜像
→ 测试环境部署
→ 微信端到端回归
→ 人工审批
→ 生产灰度
→ 健康检查和回滚确认
```

生产环境必须固定 Git commit 和镜像 digest，不使用浮动 `latest`。

## 19. Git 和上游同步策略

- Fork `tgoai/tgo` 到自有 GitHub 组织或账号。
- 本地配置 `origin` 指向自有 Fork，`upstream` 指向 `tgoai/tgo`。
- `main` 保持为稳定集成分支。
- 所有开发使用语义明确的功能分支。
- 上游更新通过独立 `chore/upstream-sync-*` 分支和 PR 合并。
- 上游同步后必须执行微信、Agent、RAG、Workflow 和人工客服回归。
- 不使用 `git reset --hard` 覆盖用户已有修改。
- `.env`、真实密钥、媒体、日志、数据库和运行缓存不得提交。

## 20. 安全和合规

- 明确提示客户正在与 AI 客服交互，并提供转人工入口。
- 建立个人信息、语音、图片和对话数据清单。
- 明确第三方 ASR、OCR、VLM 和 LLM 的数据处理地域及保留政策。
- 原始媒体加密存储，使用可配置删除期限。
- 手机号、地址、身份证和支付信息必须脱敏。
- 未经授权不得将客户对话用于模型训练。
- 客户内容、OCR 和网页知识视为不可信输入。
- 业务工具实行白名单、最小权限和审计。
- 退款、改价、账户变更等高风险操作由人工确认。
- 正式上线前完成中国境内适用的隐私和生成式 AI 法务审核。

## 21. 主要风险

### 21.1 微信链路稳定性

TGO 当前公开仓库仍有微信客服回复、企业微信机器人收消息和 AI 自动回复卡住等问题。第二周必须先完成文字链路验证，未通过前不投入多模态开发。

相关问题：

- <https://github.com/tgoai/tgo/issues/31>
- <https://github.com/tgoai/tgo/issues/32>
- <https://github.com/tgoai/tgo/issues/37>

### 21.2 上游升级冲突

通过薄改 `tgo-platform`、模块化新增能力、固定 commit 和独立升级 PR 控制风险。

### 21.3 模型误判

通过结构化输出、离线评测、置信度阈值、敏感意图强制转人工和影子运行控制。

### 21.4 提示词注入

客户文本、OCR 和图片文字不得作为系统指令；工具必须白名单、参数校验和最小权限。

### 21.5 数据和成本

记录 ASR、OCR、VLM 和 LLM 用量；相同媒体按哈希缓存；图片压缩后识别；设置超时、预算和备用模型。

## 24. 第一阶段执行顺序

1. 初始化本地项目和 Git 仓库。
2. Fork 并下载 TGO 源码。
3. 阅读 TGO 根目录和各子模块文档、依赖、迁移与启动脚本。
4. 固定验证用 commit，记录上游地址和版本。
5. 启动原版 TGO，建立基线测试结果。
6. 验证微信客服文字接收、AI 回复和人工回复。
7. 修复文字链路后再实施多模态。
8. 完成意图、Workflow、RAG 和业务工具。
9. 完成影子运行后逐步开放自动回复。
